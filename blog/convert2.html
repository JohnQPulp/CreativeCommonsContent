This is the third in a series of posts explaining my processes for how I convert books into visual novels. See the full series links over on the website's README.md.

In this post, we'll go into the first "edit" of the conversion process, wherein we create the starter pulp.txt file with its starter metadata annotations.

This is the step where we'll leverage LLM-based text processing to the maximum extent possible, automating what can be automated.

If you want to skip ahead to just the specific commands and LLM instructions used in the process, you can start reading from the The Feeding Pipeline section. But before getting there, these sections will help explain why the pipeline and instructions and set up the way that they are:

What Can't Be Automated

As described in the Part 0 post about the breakdown of all the metadata aspects of the pulp.txt format, there are a lot of metadata tags that we'll end up using throughout the full conversion process. But, not all of them make sense to LLM-generate.

As described in more detail in constraints blog post, we do not want the LLM to be in charge of assigning backgrounds. The difficult constraints that backgrounds operate under with respect to the text and with respect to themselves is simply too context-demanding a task for even the best LLMs, at least at the moment. The generated results simply wouldn't be good and would require more time to manually fix through editings and reworking as compared to assigning the backgrounds manually, thinking through at each one a rough idea of what the backgrounds will need to look like, and generating them from there. Course, even that process still ends up requiring rather significant re-editing and re-working and re-thinking, but still less so than trying to reason through LLM-outputted results. Basically, LLM output won't save us any time here as compared to just annotating the background tags from scratch.

So, we will not instruct our LLM converter to generate any b background tags or its related r reset-to-background tags. And by "not instruct", I mean we just won't even tell the LLM that these tags exist in its eventual boilerplate prompt.

And the same will apply to many other of the metadata tag types too, whose reasons for not being good automatic-generation targets fall into one of three categories:

First, if a tag's metadata is dependent on knowing what the visuals are going to look like, we can't generate it at this stage, since we obviously don't have the actual images yet. This rules out the c character tags, because we can't reason about whether a character sprite should be active if we don't know whether the background they'll be against even has a foreground. It also rules out the v viewport tag (for sizing active sprites) and the z zoom tag (for sizing characters relative to each other and to the speaker box).

Second, we don't want to automatically generate any of the tags whose use cases are too niche. The reasoning is that, the more rarely used a tag is, the higher the ratio of false-positives to good-outputs we'll end up seeing. Remember that the whole point of automatic generation is to save us work in the conversion process, but if the LLM spits out a bunch of bad metadata attributes that we then have to review and clean up, we'll end up just creating more work for ourselves. This becomes less true the smarter the LLMs become, but as of time-of-writing, I think can fairly be applied to all these tags:

  • The o object tag only comes up a handful of times per book, and has the complexity involved in reasoning about visuals and visual relationships. Therefore, the noise value of generating would be high.
  • The a age and x extra tags for modifying sprites are also used relatively infrequently, and also involve reasoning about how those changes will affect sprite appearances. This is in theory somewhat less complex of a task, but still presents the potential problem of over-annotating, introducing too many sprite variations that aren't really needed, those complicating generation, forcing either more time than necessary to be spent there, or more time spent in metadata cleanup.
  • The t thinker tag is the rarely-used cousin tag to the s speaker tag. In typical books, character thoughts presented as dialogue are infrequent enough that this isn't work muddying the waters of potentially generating t tags where s tags should go.
  • The f filter tag for applying image filters like blur() or brightness() to images is also not too common, and often requires knowing what images will look like beforehand. Often, but not always, surprisingly, since we can reason that if a background is being applied in a flashback context that it can make sense to apply a blur to it, even before knowing what the background actually looks like. Still, that's not universal, nor is it a tool worth giving the LLM to generate.
  • Lastly, the m modifier and h header tags, which affect the Pulpifier processing in various ways both have high enough complexity-to-use-case ratios as to be worth skipping.

And lastly, we generally want to avoid generating tags whose metadata need to be carried over between chapters. The reason for this has to do with the fact that, in the course of processing, we break up the book into chapter-lengths, and process those one-at-a-time. We do not feed any of the other chapters' text of any of the previously generated pulp.txt metadata into the LLM when doing the generation. This may seem like a bad idea, but the reasoning behind this is explored in more detail in the Text Feeding Approaches section further below, and it has to do with what works well with current-generation LLMs and more importantly, what tags we're dealing with in the generations. But the important point for here is that it means that, even if e.g., backgrounds were something we wanted to try generating, we'd have to do additional scripting work to carry over the background tag values between chapters. And this applies to all metadata tags whose values can span chapters, which further disincentivizes generating many of them.

As you can see, we've eliminated almost all the tag types in the above qualifications. But that's types, not volume. And the two most commonly used tags are very much automatable, so let's discuss what metadata there is left to generate:

What Can Be Automated

Flipping the script from above, the tags we do want to generate are those that are common, easy to verify and/or clean, and which are the least complex to understand. (Since again, less complexity = fewer mistakes = less manual cleanup.)

This leaves three tags to generate, and one additional non-tag aspect of the metadata. Going from least-to-most common:

The (n)ame Tag

Every character we want to assign sprites too (which in general, is usually roughly the same set as the characters with speaking roles in text) needs a n metadata tag to assign the character's ID and speaker display name.

This is the least common of the tags to generate, since in general most characters only need one name declaration. Some might need more than one if the know parts of their name change over the course of the story (or if their name itself just changes over the course of the story through e.g., marriage), but even with this bit of additional complexity, the LLM-generated results for this are still quite good. Not fully free of needing cleanup, especially sometimes around some of the minor characters, but still good enough that the cleanup is less time than assigning the names manually.

Now, previously I said we don't want to automatically generate metadata whose values need to carry over between chapters. And the name metadata certainly does, or rather its IDs do, since we need those in any chapter that that character appears. And because the idea is that we're going to be generating one chapter at a time, this means we'll need to do a bit of extra manual work between each chapter to keep the name metadata list provided to the LLM complete. However, because name tags are uncommon, and also easy to validate, and also easily understood by the LLM, this isn't a big deal, as compared to if we tried to do the same with background tags: which are way more common, and much more likely to change, and much more likely to be misinterpreted by the LLM vs what we intended.

The (s)peaker Tag

Next, we'll want the LLM to automatically assign s speaker tags, using the name IDs generated and continually fed in from above, to attribute the correct characters to each line of dialogue. (And then also to set the speaker back to none whenver dialogue ends.)

In general, the LLMs are very good at this task, tending to make fewer mistakes than a human would over the course of a long book. Fewer, but not none. And it is interesting to note that, like a human, they tend to make more mistakes when fewer dialogue cues are given, such as when authors have long blocks of back-and-forth dialogue unattributed from any "said"s, where you just have to keep track of who the speaker is by virtue of it being an even or odd paragraph.

So proofreading is a necessity here, for making sure the speakers come out correct. But the good news is that the overall conversion process requires at least two subsequent full manual re-read edits anyway, and thus we get a good level of manual double-checking more-or-less baked in.

We can also consider, for trickier texts that are more likely to have attribution mistakes, feeding the outputted pulp.txt back into another LLM, and have it double-double-check. Although in my experimenting with this, while it does work to catch some mistakes, in tends to throw up more false positives to sift through, so I wouldn't consider this an essential part of the process.

The (e)xpression Tag

The last tag we want the LLMs to automatically generated is the e expression tag.

In theory, the expression tag should be at most as common as the number of speaker tags + thinker tags + active characters in the character tags. However, there's not really any downside to overproducing on our expression quotas, since an expression tag that doesn't cover any visible sprites just does nothing. Therefore, in our eventual LLM instructions, we won't say anything about limiting the expression to only when characters are on-screen, since the LLM isn't going to know the full scope of that anyway (it doesn't know what the thinker or character tags will be set to yet). This gives us flexibility to add and shuffle tags around later on, while still having a good basis for keeping the expressions reasonably accurate. The only downside is some extra level of unnecessary expression tag noise in the generated pulp.txt output, but this isn't a big deal.

When it comes to generating expressions, any expression value is valid in terms of what the pulpifier accepts, whether it's set to grinning or gesticulating or genuflecting or whatever. But we don't want the LLM output to generate any arbitrary emotion — we want to constrain it to a pre-determined list of semi-automatically generatable emotions. This helps both make the emotions that are set easier to generate, and also means there will be fewer of them, since we'll be more likely to see sprite reuse when constrained to a smaller set.

But we don't want the set to be too small, since we still want the sprites to portray a wide range of emotions. That current set that we instruct the LLM to pull from is this:

  • neutral
  • happy
  • unhappy
  • angry
  • wary
  • nervous
  • upset
  • suspicious
  • confused
  • thinking
  • serious
  • smirking
  • weary
  • fearful
  • surprised
  • annoyed
  • scowling
  • excited
  • contemplative
  • crying

This list can easily be changed to add more expressions if emotional gaps occur, and can also be shrunk if too similar of emotions are being generated, causing essentially duplicate sprites.

So how well does this work in practice in terms of emotional accuracy? Well there's a couple aspects to consider:

First, the LLM isn't perfect about restricting the emotions it generates to just the above list. However, that's perfectly okay. A one-off special emotion to capture some special unique state can even be desirable, since that's exactly what we might also want to add in later in subsequent manual edits. And for those one-offs that we don't end up needing, it's easier to delete emotions or quickly swap them out than it is to write them up manually.

Second, the accuracy of the emotion labeling is generally pretty good, but those are just the labels. The actual sprites that get generated for those emotions (and there can be more than one sprite per emotion per character+age+extra) might differ somewhat in the intended meaning from the LLMs labeling. This comes down to the selections we do in generating the sprites, later on. We do want to disgard and regenerate sprites that come out particularly bad, but in general, sprites emotions can be a little loose in matching their labels, since the interpretation of the emotions themselves can be fairly loose. Unlike speaker attributions, which have one correct answer that needs to be verified, there's no one right answer to what expression a character should be wearing on any given slide, and so we have a lot more leeway here in assignments. There can certainly still be wrong assignments, but we often can't say whether a particular expression is right or wrong at a given time until we can actually see it in its proper context, after the sprites and backgrounds have been generated. In that sense then, expression tags are a case where it's actually faster and easier to the LLM generate a close-enough initial draft of the metadata, which we can then fix and tweak as needed later on. For such a common tag, where loose interpretation is allowed, this saved a lot of time vs trying to do all the initial expression tagging ourselves. Either way, editing will need to follow afterwards, so it's fine to let the LLM have the first pass.

Line Breaks

Lastly, there's the insertion of line breaks themselves (and the subsequent necessary adjustment of quotes, as detailed in Part 0).

This is similar to expression attribution in that, doing all the splitting ourselves would be highly tedious, as compared to just letting the LLM have the first pass, and making adjustments as needed. Like with the expression tags, there's no one right answer on where all line breaks should be inserted, although there are plenty of heuristics for where good line breaks should go, but we easily feed those into the LLM's instructions too.

There are definitely wrong places for line breaks to be inserted, such as between two random words with no puncutation separating them, but this rarely happens, because we instruct the LLM not to do that, and also even when it does happen, this is the kind of mistake the Pulpifier can automatically check for when comparing with the original book.txt source. If it finds a misplaced line break (or bad handling of quotations across line breaks, or some other line break related mistakes), it can just throw an exception, and be fixed up via manual review).

Ultimately, some manual adjustments around the breaks is always needed over the course of the editing process, and again sometimes the specific breaks we want don't become fully known to us until we can actually see all the visuals. But for the most part, this is another situation where letting the LLM do the first draft gets us 90+% of the way there, saving lots of manual editing time.

Text Feeding Approaches

We've established what all metadata we want the LLMs to generate, and we know that we'll have to feed the book text into the LLM somehow to get the metadata back out.

The most obvious approach for how to do this then would be to feed in the full book.txt text, and ask the LLM to output the full pulp.txt. However, this doesn't work.

On the input side, current LLMs can handle book lengths texts just fine. One way you can verify this is to feed in an entire book, ask a specific question, one detailed enough that the text needs to be referenced directly, and observe the (usually correct) result it spits out. In fact, this ability to quickly answer any question about a book's texts by scanning the whole book comes in very handy later on when it comes to checking character and background descriptions and image representations for mistakes.

The problem is on the output side. Even if we consider only a novella-length book of 30k words, at the current generally accepted ratio of 4:3 for tokens:words, that's 40k tokens just to spit the text back out. And we need the LLM to spit the text back out, because formatting the metadata and line breaks within the text is part of the LLM's job. Line breaks themselves are additional output tokens, as are all the new metadata annotations, though I'm not sure how high exactly this pushes up the output ratio to.

But what matters is that, from observations in practice of seeing what the top-tier LLMs can handle (and we need the smartest LLMs for the greatest accuracy, even if ones with longer output tokens exist), the ability to accurately spit back out the text with good line breaks and metadata annotations does drop as output text lengths increase. And eventually, the text either goes completey awry, or the LLM just errors out.

We can re-roll bad text generation outputs, but ideally we want to do this as infrequently as possible, and as the requested output length increases, the probability of eventually ending up with bad results approaches… if not one, then something close enough to it.

Now, the next thing to try would be to feed in the whole book text, but only ask it to convert one chapter at-a-time. Then, feed the previously converter chapter metadata (the rolling pulp.txt generation) back into the LLM, either replacing the book.txt portions already converted, or else just adding to them. The point of feeding the already-converted metadata back in is that we need the character IDs generated by previous chapters' name tags in order to correctly add additional name, speaker, and expression tags.

Overall, this approach works better, but the bad generation frequency is still too high, and the results still subpar. Even though the LLM can ingest the whole book, and even the whole book repeated again with added metadata annotations, there's still quality drop-off with the increased input context. It dilutes the LLM's capabilities.

Third attempted approach then: feed in just the particular chapter text an LLM needs at any given time, plus all previously generated metadata-annoted text. This works better still, but still starts to collapse in later chapters of longer texts, as the full weight of the ever-growing pulp.txt text crowds the input context. While the LLMs can very well understand the pulp.txt in small doses, observations with the current ones show that they really start to choke the more that gets fed in.

But the good news is that we don't really need to fill in all the previous metadata. Remember, there are four things we want the LLM to do: of these, the line breaks don't require any previous context, and the three metadata tags just need the small set of previously declared name tags. Therefore, rather than feed in the full pulp.txt with all its superfluous breaks and speaker attributions and expressions, we can just grep out just the name tags, append just that small text set to our prompt for each chapter, and then feed the chapters in one-at-a-time.

This, finally, works pretty well, with the results outputted being fairly accurate on average in terms of adherence to desired line break formatting, correct speaker attributions, correct strict adherence to the source text, and good expression attributions.

And, even when the results aren't good, doing things one chapter a time makes re-rolling for better results much easier. Though the probabilistic nature of LLMs means that we can't depend on them to always spit out good results, it also means that re-running a prompt with the exact same input can magically convert bad results into good ones.

Now, a valid objection to all this might be: if we're only feeding chapters in one-at-a-time, can we really expect the LLMs to assign accurate emotions to the characters, not knowing the full context of what came before in the story? You could even argue it might hurt the speaker tag generation accuracy as well, without the full context of how the character is referred too and tends to speak.

The answer is that, in practice, this doesn't seem to be much of a concern, for a few reasons.

First, the accuracy benefits of having smaller input and output context sizes pretty well dwarf any other effects in terms of getting the most accurate results. Again, this is just the way the current limitation of current LLMs seem to be, and it could very well be that in the not-so-distant-future this limitation goes away. But for now, it's the biggest text limitation we have to work around.

Second, as far as expressions go, the outputs we want are so heavily heuristic-driven anyway that any subtleties lost in cross-chapter emotional statuses are almost certainly going to have insignificant effects in the overall expression output quality, especially when the almost always the expression you really want is found in the immediate text at-hand of each particular chapter. And with expressions being pretty fuzzy anyway, combined with subsequent tweaks being necessary no matter what once we actually see what the generated sprites look like, being "good enough" at this stage is plenty good enough. On the other hand, speaker attribution has much more rigid right-and-wrong answers, but in practice, the LLMs seem plenty good enough reasoning just chapter-by-chapter to correctly label the speakers, even when their full names aren't restated in each chapter. Partially this is down to us feeding in the full names as part of the name attribution metadata, and part of it comes down to:

The third factor: for the types of stories being processed on Public Domain Pulp — the literary classics — the LLMs essentially have a good understanding of the overall story to begin with, even without the source text being fed in. This is demonstratably true by virtue of working with the image generators: ask them to generate a character sprite of a character from a book like The Great Gatsby without any particular details, and they'll tend to do a reasonably accurate job of it. Not as good a job as when the specific details we want are packaged into the prompt, mind you, but certainly good enough that I have no worries that, when it comes to doing the text processing, that the LLMs will somehow lose the plots of the stories they're converting when they're only allowed to see one chapter at-a-time.

An interesting question is: would this be more of a problem for if we wanted to convert lesser-known works? In practice, I still don't think so, since I think the LLM would still be able to get good results out of the chapter-length contexts still given to them for the limited set of metadata we want back out. But, without any real obscure texts attempted yet, it is still somewhat of an open question.

The Feeding Pipeline

We've established the pulp.txt format and its metadata attributes in detail.

We've established what specific subset of the metadata we want out from this initial conversion process.

We've established the overall strategy we're going to use for this conversion process: feeding in one chapter at-a-time, along with a rolling list of the previously generated name tags.

Now, we're finally ready to actually do the conversion.

Prepping the Source Text

The first thing we need to do is split up the source book.txt into chapter-length segments. To have an example to work with, we'll use Dr. Jekyll and Mr. Hyde as our example source text.

The bash command to split up the chapters looks like this:

csplit -z -f 'chapter_' -b '%02d.txt' book.txt '/^## /' '{*}'

Where we're using ## as the chapter header matcher, since that's what that particular book uses. But note that other books might require three pounds, or either two or three pounds, or some other handling, to get the splits. We might not even necessarily want to break on every possible chapter/section divider for some books, if the outputted chunk lengths are too small in their finest form. We can see for example that the chapter lengths for this basic split of Jekyll and Hyde produced some wildly different word count lengths, some of which are very short:

$ wc -w chapter_*.txt
    13 chapter_00.txt
  2402 chapter_01.txt
  2939 chapter_02.txt
   800 chapter_03.txt
  1669 chapter_04.txt
  1638 chapter_05.txt
  1496 chapter_06.txt
   555 chapter_07.txt
  4369 chapter_08.txt
  2800 chapter_09.txt
  6942 chapter_10.txt
 25623 total

Although there's an order of magnitude difference between the longest and shortest chapter here (note that chapter_00.txt just contains the book header, and so doesn't count), that's mostly fine for the purposes of what we're doing — we can still just feed things in one-at-a-time as-is. But note that we could always combine smaller chapters or split larger ones if we wanted a more strictly even breakdown. While there's no hard limit for these inputs, I think maybe ~10k words is where I'd consider stepping in to split things, so as to avoid the earlier-mentioned accuracy problems of long inputs and outputs.

Feeding the Chapters

Now, we start feeding in chapters, staring with chapter_01.txt. We do this alongside the long boilerplate prompt created for this initial conversion process, which I've included the current version of as reference in the Github repo: https://github.com/JohnQPulp/CreativeCommonsContent/tree/main/reference/vn-prompt.txt.

The prompt itself is pretty long, almost nine-thousand words, but input tokens are relatively cheap, and we need the long boilerplate to include the full explanation as to what the task is, the details of how exactly we want the line-splitting to happen, what the three types of metadata are, how to generate each, and then a long example of book.txt input paired with the draft pulp.txt output. And then finally: the title of the book we want, the current running list of characters (empty for the first chapter), and the chapter header start we want.

Then we can finally let it roll, copy append the resulting output into our rolling pulp.txt file, and begin the examination process.

The Failure Modes

There are several ways the conversion process can go wrong. As mentioned previously, the probability of it going wrong is some function of the chapter length, but not a linear one, and bad results can come from any length of chapter, even if less likely from the shorter ones.

Re-rolling the output for better results is the go-to response if we encounter any of these worst failure modes:

First off, the most entertaining conversion problem that can happen in this process is that the LLM will lose the plot of the original text it's meant to be converting, and just start writing its own continuation of the story. It seems the the LLM can usually recover from simple one-word transcription errors, but sometimes these spiral into just complete fabulizing off of the source text, essentially turning into fanfiction, or I suppose an alternate retelling of part of the story. It's funny that these "spinoff" sequences tend to markedly poorer written that the original text, which I suppose on one hand makes sense, since they're up against the greats of the classics, though on the other hand I would expect the smartest LLMs to do better at creative writing at this point, though on the other other hand, I suppose they are being asked to do two things at once by that point, since they still also tend to spit out metadata against their fake story continuations.

The other main big failure mode is problems around line breaks. Sometimes the LLM will decide to only insert a single line break between two text lines without any metadata, which is incorrect, since we want a 3-cycle sequence. Other times they'll insert one extra line break, turning it into a 4-cycle. And other times still, they'll fail at the quote-handling part of the line breaks, not adding new quotes where they're necessary for the pulp.txt format. These are all fixable mistakes given effort, but we want to be able to put in the minimal effort, and so the best approach is just re-rolling.

Then, there are the smaller mistakes:

As mentioned above, sometimes the LLM just transcribes one particular token wrong. Often this is the LLM "fixing" what it sees as a mistake, like normalizing an uncommon spelling. This isn't what we want, but such mistakes are easily identified by the Pulpifier in its text comparison between the original text and the new output, and so point mistakes like that can just be manually fixed.

Another thing the LLM likes to do is split up lines via semicolons, which is usually not where we want line breaks. Though we sometimes want it, which means that the Pulpifier does allow line breaks via semicolons, which means these don't show up as compilation errors. However, we can pretty easily remove all the semicolon-based line breaks with some find/replace regexes in gedit.

This one captures line breaks on lines ending in semicolons, are recombines both the text lines into one and metadata lines into one:

   Find: \n\n([^\n]*;)”?\n([^\n]*)\n\n“?([^\n]*)\n([^\n]*)
Replace: \n\n\1 \3\n\2;\4

Combining the metadatas into one can lead to some new attribution mistakes, so the replace results from this do need to be quickly manually checked, though usually there aren't too many to go through. Merging the metadatas also results in there being lots of redundant semicolons, but these can be cleaned up with more replaces:

   Find: \n;
Replace: \n

   Find: ;\n
Replace: \n

   Find: ;;
Replace: ;

Note that for all of these, you'll want to hit the "Replace All" button multiple times until it de-highlights, since doing these replaces can create new cases to replace.

The conversion process can also put line breaks in places that are valid, but just suboptimal for how we want the VN to be ready. Sometimes this is having lines that are too long, or too many lines in-a-row that are too short, or lines that combine multiple settings, or lines that combine dialogue and non-dialogue. Part of this initial first edit process requires scanning the outputted results anyway, and if mistakes such as these are noticed, they can be quickly fixed, though in general it's fine to leave these to the second edit.

Speaker attributions are somewhat similar, although their mistakes can be a little trickier to identify if left for later, since we'll often end up needing to go back and reference the original source text to see how the quotes were formatted there to know if back-and-forth dialogue is labeled correctly, since we lose the original paragraph markers. But at least for speakers, mistakes tend to be much rarer, and usually still catchable in the second edit, or if not then, then the third.

Name attribution mistakes essentially never happen, though sometimes we'll want to tweak the display names of characters, or remove redundant name declarations, or add in additional ones for renaming, so as to not reveal a character's full name right away, or for whatever other reason. Since name tags are the least common, and the most important to get right, since we'll be feeding them forward, we don't want to review all of them individually, correcting them if needed, as part of the first edit.

This regex can be used to quickly round up all the declared names:

grep -Eo 'n:[^=]*=[^;]*' pulp.txt

Note that throughout all this, we'll want to keep checking the compilation status of the new pulp.txt contents by running the PulpifierCLI tool against it. This identifies all the errors that can be automatically identified. Assuming the chapter has no remaining compilation errors within itself, the PulpifierCLI will still show an error if the chapter isn't the last chapter, showing a mismatch on the expected header of the following chapter. And that's how we know that the chapter is fully Pulpifier-ready.

Feeding Forward

After we've gotten our chapter all nice and converted, we move on to the next one. But before we try feeding it in, we need to update the rolling character list. We can use the same regex sa included above for the name tag cleanup to pull the names out, and then copy them into the end of the boilerplate prompt, along with updating the desired target chapter header, and selecting the next chapter file for our conversion.

Note that we don't necessarily want to carry every last character over between chapters, since some of them might be one-offs that have no bearing, and might have generic identifiers like waiter or policeman that we wouldn't want to get reused in subsequent chapters. (In fact, it's often beneficial to swap the IDs for these to ones like waiter1 or policeman1 to help make sure they don't get accidentally reused between chapters.)

But those minor prompt tweaks aside, it's basically just the same process again, and again and again, for each chapter, until we've gotten the whole book's pulp.txt out.

All Done (With the First Edit)

And with that, we've gotten our starter metadata-annotated text to build off of.

Overall, while the process takes a lot of explanatory setup to show why it is as it is, the actual conversion itself is not so bad, with the first edit being by far the shortest of the edit steps, or of any of the steps really. It just has a lot to talk about.

But while the second edit, where we finish the metadata rough draft by adding in the rest of the metadata tags, is a much longer process, since it's all manual, it's conceptually rather simpler, even if it does hold some interesting discussion topics.

But the explanation will have to wait for the next part: The Second Edit.
